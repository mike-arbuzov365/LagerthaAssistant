# PRD: BaguetteDesign — Telegram-бот для графічного дизайнера

> Версія: 1.0 | Статус: Фінальний |Scope: V1 MVP
> Джерело: detalnyi_plan_v1_professional.pdf + дослідження

---

## Цілі продукту

| Ціль | Метрика успіху |
|---|---|
| Клієнт отримує відповідь 24/7 без дизайнера | 80% питань закривається автоматично |
| Дизайнер витрачає ≤30 хв/день на комунікацію | Вимірюється через ConversationIntentMetrics |
| Жоден лід не губиться | 100% звернень зберігаються в БД |
| Scope creep під контролем | Лічильник правок активний на всіх проєктах |

---

## Ролі

### Клієнт
Людина, яка хоче: дізнатись про послуги / подивитись портфоліо / уточнити ціну / пройти бриф / передати запит / дізнатись статус.

**Ідентифікація:** будь-який Telegram user крім `ADMIN_USER_ID`.
**Вхід:** прямо або через реферальний лінк `t.me/bot?start=ref_xxx`.

### Дизайнер (власник бота)
**Ідентифікація:** `tg_id == ADMIN_USER_ID` (з environment variable).
**Можливості:** повний доступ до inbox, лідів, проєктів, налаштувань.

## User Stories та Acceptance Criteria

### US-01: Задати питання

**Story:** Як клієнт, я хочу поставити довільне питання про послуги, щоб отримати відповідь без проходження брифу.

**AC:**
- [ ] Claude відповідає на будь-яке питання про послуги протягом 5 секунд
- [ ] Після відповіді бот пропонує 3–4 наступних кроки кнопками
- [ ] Якщо питання поза компетенцією AI — пропонує зв'язатися з дизайнером
- [ ] Контекст розмови зберігається між сесіями (UserMemoryEntries)
- [ ] Голосові повідомлення транскрибуються (V2)

**Out of scope V1:** Транскрипція голосу

---

### US-02: Бриф (центральна функція)

**Story:** Як клієнт, я хочу пройти бриф у форматі живого діалогу, щоб передати всі потрібні деталі проєкту без заповнення анкети.

**AC:**
- [ ] Бот ставить питання послідовно, наступне залежить від попереднього
- [ ] Питання адаптуються під тип послуги: логотип → питання про бренд; Instagram → питання про кількість і формат
- [ ] Клієнт може пропустити питання, повернутись назад, зупинити і продовжити пізніше
- [ ] Клієнт може надіслати файли/зображення прямо під час брифу
- [ ] AI перевіряє повноту: якщо немає критичних полів — запитує повторно перед завершенням
- [ ] Клієнт бачить summary брифу і підтверджує перед відправкою
- [ ] Після підтвердження: лід автоматично створюється в БД + сторінка в Notion
- [ ] Дизайнер бачить у inbox: ім'я, тип послуги, summary, позначку повноти

**Conversation flow:**
```
Bot: Який тип дизайну вам потрібен?
     [Логотип] [Айдентика] [Соцмережі] [Пакування] [Інше]

User: [Логотип]

Bot: Розкажіть про ваш бізнес. Як називається компанія і чим займаєтесь?

User: "Кав'ярня 'Ранок', продаємо спеціалітет-каву в Києві"

Bot: Які цінності та емоції має передавати логотип?
     (можна написати або обрати): [Затишок] [Енергія] [Преміум] [Свій варіант]

... (ще 6-7 питань адаптивно)

Bot: Дякую! Ось що я зафіксував:
     • Тип: Логотип
     • Бізнес: Кав'ярня "Ранок", Київ
     • Цінності: Затишок, якість
     • Стиль: Мінімалізм
     • Дедлайн: 2 тижні
     • Бюджет: $200-300
     
     Надіслати дизайнеру?
     [✅ Надіслати] [📎 Додати матеріали] [✏️ Редагувати]
```

**Out of scope V1:** Spec.md генерація (є в V2)

### US-03: Прайс

**Story:** Як клієнт, я хочу побачити вартість послуг за категоріями, щоб розуміти бюджет перед брифом.

**AC:**
- [ ] Бот показує категорії: логотип / айдентика / соцмережі / поліграфія / веб / інше
- [ ] Після вибору категорії: діапазон вартості + що входить + від чого залежить ціна
- [ ] Є кнопка переходу до брифу для точнішої оцінки
- [ ] Дані прайсу беруться з Notion DB (дизайнер оновлює без деплою)
- [ ] Різні ціни для ринків: Україна / USD / EU — автоматично по країні клієнта

---

### US-04: Портфоліо

**Story:** Як клієнт, я хочу переглянути приклади робіт за категоріями, щоб зрозуміти стиль і прийняти рішення.

**AC:**
- [ ] Категорії: логотипи / айдентика / соцмережі / пакування / поліграфія
- [ ] Кожен кейс: прев'ю + короткий опис + для якої ніші
- [ ] Кнопка "Хочу схожий стиль" → бриф з pre-filled полем стилю
- [ ] Дані з Notion DB (без деплою)
- [ ] Посилання на повну добірку (Behance або сайт)

---

### US-05: Зв'язатися з дизайнером

**Story:** Як клієнт, я хочу зв'язатись з дизайнером або запланувати дзвінок, не виходячи з Telegram.

**AC:**
- [ ] Три варіанти: Написати повідомлення / Передати бриф / Запланувати дзвінок
- [ ] Якщо без брифу — м'яка рекомендація заповнити спочатку
- [ ] Дзвінок: бот показує вільні слоти → клієнт обирає → подія в Google Calendar з посиланням Zoom/Meet
- [ ] Після запиту клієнт отримує підтвердження з очікуваним часом відповіді
- [ ] Дизайнер отримує в inbox: ім'я, запит, AI summary переписки

---

### US-06: Статус проєкту

**Story:** Як клієнт, я хочу бачити простий поточний статус мого запиту або проєкту.

**AC:**
- [ ] Команда `/status` або кнопка з головного меню
- [ ] Людські статуси (не технічні): Запит отримано → Бриф перевіряється → В роботі → Чекаємо матеріали → На погодженні → Правки → Завершено
- [ ] При зміні статусу дизайнером клієнт автоматично отримує повідомлення
- [ ] Показує що потрібно від клієнта (якщо статус "Чекаємо матеріали")

---

### US-07: Inbox дизайнера

**Story:** Як дизайнер, я хочу бачити всі звернення в структурованому списку з контекстом, щоб відповідати ефективно.

**AC:**
- [ ] Список чатів зі статусами: Нові / В роботі / Очікує відповіді / Закриті
- [ ] Кожен елемент: ім'я, тип послуги, AI summary, час останнього повідомлення
- [ ] При відкритті чату: повна переписка + AI summary + поля брифу + кнопки дій
- [ ] Гібридний режим дефолтний: AI готує чернетку → [Надіслати] [Редагувати] [Відхилити]
- [ ] Внутрішні нотатки (клієнт не бачить)
- [ ] Шаблони відповідей для типових ситуацій

---

### US-08: Ліди

**Story:** Як дизайнер, я хочу бачити структуровані заявки окремо від діалогів, щоб вести їх по воронці.

**AC:**
- [ ] Картка ліда: ім'я, контакт, тип послуги, країна, бюджет, джерело, дата, статус, повнота брифу
- [ ] Статуси: new / negotiation / contract / rejected
- [ ] Дії з картки: відкрити чат / змінити статус / створити проєкт / надіслати КП / запланувати дзвінок / нотатка
- [ ] Автосинхронізація з Notion DB (ліди видно в Notion)

---

### US-09: Проєкти

**Story:** Як дизайнер, я хочу вести активні проєкти з контролем статусу, матеріалів і правок в одному місці.

**AC:**
- [ ] Картка проєкту: назва, клієнт, тип, статус, дедлайн, бриф, файли, коментарі, правки
- [ ] Статуси: Бриф → Договір → В роботі → Правки → На фіналі → Здано
- [ ] Двостороння синхронізація з Notion: зміна статусу в Notion → повідомлення клієнту
- [ ] При створенні проєкту → авто-структура папок у Google Drive: `Projects/2026/ClientName/ProjectTitle/[Source|Final|Revisions]`

---

### US-10: Лічильник правок

**Story:** Як дизайнер, я хочу автоматично рахувати кола правок, щоб контролювати scope creep і мати підставу для доплати.

**AC:**
- [ ] Картка проєкту показує: "Правки: 1 / 2" (використано / включено в пакет)
- [ ] Дизайнер може зафіксувати нове коло правок з описом
- [ ] Клієнт бачить: "Використано 1 з 2 кіл правок, що входять у пакет"
- [ ] При перевищенні ліміту — дизайнер отримує сигнал з пропозицією повідомити клієнта

---

### US-11: Комерційна пропозиція

**Story:** Як дизайнер, я хочу швидко генерувати КП на основі брифу, щоб не витрачати час на написання з нуля.

**AC:**
- [ ] Кнопка "Створити КП" в картці ліда або чату
- [ ] Claude генерує чернетку: опис задачі / обсяг / терміни / вартість / умови
- [ ] Дизайнер редагує і надсилає клієнту в Telegram
- [ ] Копія автоматично зберігається в Notion

---

### US-12: Файли та матеріали

**Story:** Як дизайнер, я хочу бачити всі матеріали від клієнта прив'язаними до конкретного проєкту/ліда.

**AC:**
- [ ] Клієнт надсилає файли прямо в Telegram → бот прив'язує до активного проєкту/ліда
- [ ] Авто-завантаження у Google Drive у папку "Raw Materials" відповідного проєкту
- [ ] Авто-теги: логотип / референс / брендбук / текст
- [ ] Дизайнер бачить блок "Матеріали" з датою і підписом
- [ ] Кнопка "Запросити матеріали" → повідомлення клієнту

---

### US-13: Розумні нагадування

**Story:** Як дизайнер, я хочу отримувати тригерні нагадування про важливі події, щоб нічого не пропустити.

**AC:**
- [ ] "Клієнт не відповів 3 дні" → запит до дизайнера [Нагадати] [Ігнорувати]
- [ ] "Завтра дедлайн" → повідомлення дизайнеру
- [ ] "Рахунок не оплачено 7 днів" → нагадування
- [ ] Нагадування клієнту за 24 год і 1 год до запланованого дзвінка
- [ ] Тижневий дайджест щопонеділка: нові ліди, активні проєкти, найближчі дедлайни
- [ ] Всі нагадування дизайнеру — з кнопкою підтвердження (не авто-спам)

## Що НЕ входить у V1 (явно)

- Telegram Mini App
- AI Vision перевірка макетів
- OCR аналіз тексту на дизайнах
- Екстракція палітри з референсів
- Multi-tenant (кілька дизайнерів)
- Фінансова аналітика і звіти
- Multi-agent система
- PDF-генерація документів
- Голосовий ввід (Whisper)

## Мовна підтримка

V1: українська (основна) + англійська.
Логіка: `language_code` з Telegram metadata → якщо `uk` → українська, інакше → англійська.
Дизайнер може примусово встановити мову для конкретного клієнта.
